# Design: Projections Dashboard

**Date:** 2026-06-12
**Status:** Approved
**Scope:** Card-level projection block + Watchlist projections tab + Self-improving algorithm library

---

## Overview

The Projections Dashboard turns the raw signals, mechanics scores, and price history already in the app into clear **BUY / HOLD / SELL** recommendations with confidence and plain-English reasoning. Every projection is generated by Claude (LLM synthesis) and simultaneously codified into a deterministic algorithm that the system retains and improves over time. As algorithms accumulate validated success, future projections for matching patterns skip Claude entirely.

Two surfaces:
1. **Card detail page** (`/cards/[id]`) — projection shown directly as the back face of the card's display
2. **Watchlist Projections tab** (`/watchlist` → Projections view) — gallery of flip cards; front = card face with art + price, click to flip and reveal the projection back

## Feature 1 — Flip Card Component

### Card front face
- Card art from Scryfall `image_uri` (the tracked printing's image)
- Card name in serif font (Georgia)
- Set symbol in top-right of name bar, colored by signal strength: mythic orange (8–10), rare gold (6–8), uncommon silver (4–6)
- Verdict ribbon (BUY / HOLD / SELL) in top-left of art box
- `↺ projection` hint badge in top-right of art box
- Current price + 24h delta overlay on art
- Type line: card type from Scryfall `type_line` field (e.g., "Legendary Creature — Monkey Pirate") — fetched once per card name via Scryfall API and stored in a `type_line` field added to `WatchlistEntry`. Falls back to the set name if unavailable.
- Footer: set code, collector number, finish, confidence %
- Hover: slight 3D tilt (CSS `perspective` + `rotateY`)

### Card back face (projection)
- Same frame color as verdict: green=BUY, gold=HOLD, red=SELL
- **Name bar**: "Market Projection" label + horizon indicator (4W shown active, 12W shown greyed/disabled — non-functional until 12W data is available) + refresh button
- **Chart box**: 6-month price history as SVG sparkline. Chart line and gradient fill match verdict color. Overlays: start price (top-left), current price + delta (top-right), sell window pill (bottom-left if active), month labels (bottom). PEAK annotation line when chart shows a peak-and-decline pattern.
- **Type line**: primary signal types driving verdict (e.g., "Sorcery — Format Staple · Buy Hype") + set symbol with rarity coloring for signal strength
- **Text box**: 
  - Background gradient matches verdict color (dark green/gold/red)
  - `INTEL` watermark behind text (5% opacity)
  - Keywords line: bold, key mechanics/signals (e.g., "Break Score 8.2 · Bullish · Accumulation")
  - Rules text: Claude's 2-3 sentence analysis
  - Flavor text (italic, separated by hairline): outcome timing or most important single signal
- **Footer**: source badge (⚙ algo or Claude), validation count, set + collector number, "Illus. Claude AI", refresh button
- **P/T box** (bottom-right): confidence % — styled as the classic creature stat box

### Flip mechanics
- Pure CSS 3D: `transform-style: preserve-3d`, front/back `backface-visibility: hidden`
- Transition: `transform 0.55s cubic-bezier(0.4, 0.2, 0.2, 1)`
- Toggle class `flipped` on click → `rotateY(180deg)`
- Card detail page (`/cards/[id]`) shows the back face directly — no flip needed

### Mana pip signal indicators (name bar, back face)
Colored circles in the top-right of the name bar encode which signal types are active:
- Green pip (B) — bullish sentiment
- Blue pip (F) — format staple signal
- Red pip (P) — price peak / ban risk
- Gold pip (M) — high mechanics score (break_score or format_scores)
- Grey pip (G) — general / neutral signal

## Feature 2 — Watchlist Projections Tab

### View switcher
The `/watchlist` page gains a tab row: **List** | **✦ Projections**. The existing list view is unchanged. Projections is the new tab.

### Projections tab layout
- Cards grouped into three sections: **● BUY**, **◎ HOLD**, **✕ SELL**
- Each section header: verdict color, count badge
- Cards within each section displayed as a horizontal wrapping gallery of `FlipCard` components
- Empty sell state: italic serif placeholder text ("No sell recommendations right now.")
- Section headers use monospace uppercase labels matching the app's existing style

### Data loading
On mount, the Projections tab requests projections for all watchlisted cards via `/api/projections/[cardName]`. Requests are batched (Promise.all). Cached projections (within 24h TTL) return instantly. Stale or missing projections trigger a Claude call server-side.

## Feature 3 — Card Detail Integration

On `/cards/[id]`, a **ProjectionCard** component renders below the IntelPanel. It shows the back face directly (no flip trigger needed — the user is already on this card's page). Includes the full projection: 6-month price chart, analysis, confidence, source badge, refresh button.

The 4W horizon is the default and only option for now (12W deferred — requires data we don't have yet).

## Feature 4 — Projection API

### Route: `GET /api/projections/[cardName]`

1. Check `card_projections` table for a fresh row (`expires_at > now()`)
2. **HIT** → return cached projection JSON
3. **MISS / STALE**:
   a. Load card data: `card_mechanics` (break_score, ban_risk, format_scores, sell_window), recent `intel_signals` (last 30 days), price history (90 days from `/api/prices`)
   b. Run all `promoted=true` algorithms from `projection_algorithms` whose conditions match this card
   c. If algorithm consensus confidence ≥ 0.80 → use algorithm result, set `source: 'algorithm'`
   d. Else → call Claude API with structured prompt, set `source: 'claude'`
   e. Upsert result into `card_projections` with `expires_at = now() + 24h`
   f. Upsert algorithm into `projection_algorithms` (replace if better — see below)
4. Return projection JSON

### Claude prompt structure
Input (structured JSON):
```
{
  card_name, break_score, ban_risk, ban_risk_by_format,
  format_scores, sell_window, mechanics[],
  recent_signals: [{ signal_type, sentiment, signal_strength, ingested_at }],
  price_history: [{ date, price }]  // 90 days
}
```

Required output (structured JSON):
```json
{
  "verdict": "BUY" | "HOLD" | "SELL",
  "confidence": 0.0–1.0,
  "reasoning": "2-3 sentence plain-English analysis",
  "flavor_text": "One evocative sentence (outcome timing or key signal)",
  "key_signals": ["Break Score 8.2", "Bullish Sentiment", "Accumulation Pattern"],
  "signal_pips": ["sentiment", "signal", "mechanics"],
  "algorithm": {
    "purpose_key": "kebab-case-slug-describing-this-pattern",
    "purpose_description": "Human-readable one-line description",
    "conditions": [
      { "field": "break_score", "op": "gte", "val": 7.5 },
      { "field": "sentiment", "op": "eq", "val": "bullish" }
    ],
    "verdict": "BUY",
    "confidence": 0.85
  }
}
```

Valid condition operators: `gt`, `gte`, `lt`, `lte`, `eq`, `neq`. Field names map directly to card data keys (`break_score`, `ban_risk`, `sentiment`, `signal_count`, `price_trend_7d` as `"rising"|"falling"|"flat"`).

## Feature 5 — Self-Improving Algorithm Library

### `projection_algorithms` table
```sql
id              UUID PRIMARY KEY DEFAULT gen_random_uuid()
purpose_key     TEXT UNIQUE NOT NULL        -- the "slot" identifier
purpose_description TEXT NOT NULL
algorithm_json  JSONB NOT NULL              -- if/then rule
success_rate    FLOAT DEFAULT 0            -- composite 0–1
validation_count INT DEFAULT 0
promoted        BOOLEAN DEFAULT false       -- true = can skip Claude
created_at      TIMESTAMPTZ DEFAULT now()
last_validated_at TIMESTAMPTZ
```

### Algorithm upsert logic (replace, don't accumulate)
```
No existing slot for purpose_key   → INSERT, promoted=false
Slot exists, new confidence > old  → UPSERT (replace algorithm_json, reset validation_count to 0, promoted=false)
Slot exists, new confidence ≤ old  → discard, keep existing
```

### Promotion threshold
After `validation_count ≥ 5` AND `success_rate ≥ 0.75` → set `promoted = true`. Promoted algorithms are tried before Claude on the next matching projection.

### Algorithm runner
For a given card's feature set, evaluate all `promoted=true` algorithms:
1. Check each algorithm's `conditions` array against the card's data
2. Collect all matching algorithms' verdicts + confidence
3. If ≥ 2 algorithms match and all agree → return their verdict with mean confidence
4. If algorithms disagree or < 2 match → fall back to Claude

## Feature 6 — Composite Feedback System

### `card_projections` table (additions for feedback)
```sql
outcome_price_validated   BOOL DEFAULT false
outcome_signal_validated  BOOL DEFAULT false
outcome_user_validated    BOOL DEFAULT false
outcome_score             FLOAT              -- composite 0–1
validated_at              TIMESTAMPTZ
```

### Three feedback signals

**1. Price outcome (automated, 28-day lag)**
Hourly cron (extends the existing push-notify cron) checks projections where `cached_at < now() - 28 days` and `outcome_price_validated = false`. Fetches current price vs price at `cached_at`. BUY correct if price ↑ ≥ 5%, SELL correct if price ↓ ≥ 5%, HOLD correct if |delta| < 5%. Updates `outcome_price_validated = true`.

**2. Signal convergence (automated, 7–14 day lag)**
Same cron checks projections where `cached_at < now() - 7 days` and `outcome_signal_validated = false`. Fetches new `intel_signals` since `cached_at`. If ≥ 2 new signals align with verdict sentiment → correct. Updates `outcome_signal_validated = true`.

**3. User confirmation (manual)**
After a card's projection horizon passes (28 days), the Projections tab shows a "Did this play out?" prompt on the card with thumbs up/down. POST to `/api/projections/[id]/validate` with `user_correct: boolean`. Updates `outcome_user_validated`.

### Composite score calculation
`price_correct`, `signal_correct`, `user_correct` are cast to 1 (correct) or 0 (incorrect / not yet available).

When computing the composite, normalize weights by which signals are actually available:
```
weights = { price: 0.5, signal: 0.3, user: 0.2 }
collected = only signals that have been validated (boolean)
outcome_score = sum(weight * value for collected signals) / sum(weights for collected signals)
```

Trigger composite calculation when all three are collected, OR after 35 days (use whatever is available). Update the algorithm's `success_rate`:
```
new_success_rate = ((old_success_rate * old_count) + outcome_score) / (old_count + 1)
validation_count += 1
```

## Database migrations

### `card_projections`
```sql
CREATE TABLE card_projections (
  id                        UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  card_name                 TEXT NOT NULL,
  verdict                   TEXT NOT NULL CHECK (verdict IN ('BUY','HOLD','SELL')),
  confidence                FLOAT NOT NULL,
  reasoning                 TEXT NOT NULL,
  flavor_text               TEXT,
  key_signals               JSONB,
  signal_pips               JSONB,
  algorithm_json            JSONB,
  source                    TEXT NOT NULL CHECK (source IN ('claude','algorithm')),
  purpose_key               TEXT,
  cached_at                 TIMESTAMPTZ NOT NULL DEFAULT now(),
  expires_at                TIMESTAMPTZ NOT NULL,
  outcome_price_validated   BOOL DEFAULT false,
  outcome_signal_validated  BOOL DEFAULT false,
  outcome_user_validated    BOOL DEFAULT false,
  outcome_score             FLOAT,
  validated_at              TIMESTAMPTZ
);
CREATE INDEX ON card_projections (card_name, expires_at);
```

### `projection_algorithms`
```sql
CREATE TABLE projection_algorithms (
  id                  UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  purpose_key         TEXT UNIQUE NOT NULL,
  purpose_description TEXT NOT NULL,
  algorithm_json      JSONB NOT NULL,
  success_rate        FLOAT NOT NULL DEFAULT 0,
  validation_count    INT NOT NULL DEFAULT 0,
  promoted            BOOLEAN NOT NULL DEFAULT false,
  created_at          TIMESTAMPTZ NOT NULL DEFAULT now(),
  last_validated_at   TIMESTAMPTZ
);
```

## Out of Scope (this spec)

- 12W horizon — requires tournament calendar and set release data not yet available
- Per-card notification for projection changes — deferred to push notification enhancement
- Algorithm editor UI — algorithms are managed automatically; manual review via Supabase dashboard for now
- Multi-user projection personalization — single-user architecture, same migration path as watchlist
- Historical projection accuracy leaderboard — good future feature once enough validated data exists
